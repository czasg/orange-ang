---
title: 提交规范
slug: commit
---

我们知道，git commit message 是你对本次提交的摘要。

对于你个人而言，一个简单的 commit message 可能并不会带来什么影响，但是对于团队而言，一个简陋、不可读的 commit message，完全可能就导致了一个不可维护项目的诞生。

因此，一个好的 commit message 对于一个持续维护的项目而言是至关重要的，它可以帮助项目团队成员之间快速的建立沟通渠道，包括功能迭代说明、历史修复说明等，从而提高项目的可理解性、可维护性。

目前业界通用的 commit 规范大致如下：

```text
<type>: <subject>

<body>

<footer>
```

在该规范中，定义了四个模块：

- **<type>：**提交类型，如 feat（新功能）、fix（修复 bug）、docs（文档变更）、style（格式、空格等不影响代码逻辑的变动）等。
- **<subject>：**提交简要描述，用一句话概括提交的内容。
- **<body>（可选）：**详细描述提交的修改内容。
- **<footer>（可选）：**包含与提交相关的其他信息，如关联的 issue 号等。
