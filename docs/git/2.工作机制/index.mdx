## 背景

虽然在日常工作中，我们可以依靠成熟的工具来完成 Git 操作，但了解 Git 的底层工作机制对于问题的分析与排查非常有帮助。

Git 作为最先进的分布式版本控制系统，深入了解其工作原理可以帮助我们更好地理解版本控制的概念和原则，从而更有效地利用 Git 来管理项目，并在遇到问题时能够快速定位并解决。


## 目的

学习并了解 Git 工作机制与系统架构


## 系统架构

Git 是一个分布式版本控制系统，它引入了**远端仓库**和**本地仓库**的概念。

在日常工作中，我们主要操作的是本地仓库，并通过编辑器如 VSCode/IDE 来进行 Git 管理，当然熟悉原生指令的同学也可以通过 Git 原生指令进行手动管理。

而远端仓库一般由成熟的仓库管理系统维护，例如 Github、Gitlab 等系统进行管理。

这些远端仓库提供了稳定的代码托管服务，可以让开发者方便地共享、协作和管理代码。
通过与本地仓库的交互，开发者可以轻松地将本地修改同步到远端仓库，或者获取远端仓库的最新版本数据，从而实现分布式协作开发。

在本地仓库中，我们可以进一步划分为**工作区**、**暂存区**和**本地仓库**。

- 工作区是我们日常工作的区域，我们在这里进行代码修改和开发。
- 修改完成后，我们可以将修改添加到暂存区，暂存区相当于一个缓冲区，用于存放即将提交到本地仓库的修改。（`git add`）
- 当我们完成一系列修改后，可以通过提交将所有的修改提交到本地仓库，这样本地仓库就会新增一个最新的版本。（`git commit`）

在本地仓库更新了新的版本后，远端仓库可能会落后于本地仓库，因此我们可以通过推送（`git push`）将最新的版本推送至远端仓库，以保持远端仓库与本地仓库的同步。

同样地，当远端仓库的版本高于本地仓库时，我们可以通过拉取（`git pull`）操作来获取远端仓库的最新版本数据，以确保本地仓库与远端仓库的一致性。
